Jest配置

automock

Boolean(默认值：false)

这个配置项用来配置是否将测试文件中所有引入的模块自动变成模拟的模块
如果开启的话，就测试文件中的模块的实现都会被重新实现，但会保留 Api 接口

比如我们要测试某个模块中的某个函数功能：

// mod.ts
module.exports = () => 1

// index.test.tsx
const aotoMockFn = require('./mod')

it("test1", () => {
  expect(aotoMockFn()).toBe(1);
})

此时如果我们想测试 aotoMockFn 返回2 的情况，实际情况下就我们需要去改动内部的逻辑和代码，就不太合理

jest提供了mock 模块的功能，我们将mock一下 mod模块，就可以使用mock 的一些api，比如模拟返回值 mockReturnValue：

// index.test.tsx
jest.mock('./mod')
const aotoMockFn = require('./mod')

mockFn.mockReturnValue(2)

it("test1", () => {
  expect(mockFn()).toBe(2);
})

这样就可以很愉快的进行mock 模块了

这个配置项提供的就是这个功能的全局开关配置，开启之后引入的模块就自动是mock之后的了，无须再在每个测试文件的顶部再执行 jest.mock 之类的mock方法

bail

Number | Boolean (默认值：0)

默认情况下，Jest会跑所有的用例并且在结束的时候会把所有的错误信息打印到控制台上
这个选项可以设置当n个test失败之后停止执行。
true 等同于 1

文档中所说的**n个tests，**是指文件维度而不是一个case的维度，相关的issue：

https://github.com/facebook/jest/issues/2626

cacheDirectory - draft

String (默认值：/tmp/)

Jest会先扫描测试用例的依赖树关系生成一些信息并缓存，用来减少用例执行时对文件系统的变动造成的影响
这个配置可以自定义Jest缓存依赖信息的目录

配置了之后就是生成一些临时文件，这些依赖信息缓存下来之后带来的性能提升具体也还没测过，后面可以测试下

clearMocks

Boolean (默认值：false)

每个case执行前会自动清空Jest mock生成函数的 calls、instances、contexts
相当于在每个case执行前执行一边 jest.clearAllMocks

以calls为例：

const fn = jest.fn((x) => {
  return x
})

it('1', () => {
  expect(fn(1)).toBe(1)
  expect(fn(1)).toBe(1)
  expect(fn(1)).toBe(1)
  expect(fn.mock.calls.length).toBe(3)
})

it('2', () => {
  expect(fn(2)).toBe(2)
  expect(fn.mock.calls.length).toBe(1)
})

calls.length 表示调用次数，如果开启clearMock，那么case1执行完成之后，fn被执行的次数就会清空，case2就是对的，反之不会清空，case2中fn的执行次数为4

collectCoverage

Boolean (默认值: false)

控制是否要生成代码覆盖率报告，以及控制台是否打印覆盖率的日志

这个就是开启了之后会生成代码覆盖率的日志和报告，因为每个文件执行的时候会给每个测试文件生成覆盖率收集的文件，所以会显著降低测试速度

collectCoverageFrom

undefined | Array (默认值: undefined)

设置要生成代码覆盖率报告的目录和文件
格式为遵循 glob pattern 格式的字符串数组

这里面的路径填写的是被测试文件的路径，而不是测试用例的文件的路径

coverageDirectory

String 默认值（undefined）

覆盖率报告生成的目录(基于项目根路径)

coveragePathIgnorePatterns

Array（默认值：[‘/node_modules/’]）

设置生成代码覆盖率报告要排除的目录和文件
遵循正则表达式，不支持绝对路径匹配，使用 <rootDir> 来作为项目的根目录字符串，
防止因为不同的环境造成不同的根路径，导致意外地忽略了所有的文件

coverageReporters

Array<string ｜ [string, options]>（默认值：[“clover”, “json”, “lcov”, “text”]）

生成的代码覆盖率报告的文件格式列表
支持的格式：https://github.com/istanbuljs/istanbuljs/tree/master/packages/istanbul-reports/lib

coverageThreshold

Object （默认值：undefined）

配置代码覆盖率的四个指标阈值：
- branches：分支覆盖率
- functions：函数覆盖率
- lines：行覆盖率
- statements：声明覆盖率

如果是想全局配置，就配置在global的key下，如果想单独配置某个文件的指标，key值写在符合glob pattern模式的路径key下：

![](data:image/svg+xml;utf8,)

displayName

String | Object （默认值：undefined）

用于自定义当前项目执行用例时打印的标记和颜色
因为如果一个仓库中有多个项目，可能会有多份配置文件，这样可以更容易区分

fakeTimers

String（默认值：real）

是否使用Jest内置的定时器
针对定时器，在单元测试的环境中不可能和真实的代码环境一样，所以需要使用内置的定时器来帮助我们更灵活的书写单元测试
这个配置项就是模拟一些定时器的行为，比如跑单测碰到一个定时器，不可能真的等定时器执行的时候再继续执行单测

先用个最简单的例子：

const fn = jest.fn()

function testTimer () {
  setTimeout(() => {
    fn()
  }, 3000)
}

test('test timer', () => {
  testTimer();
  expect(fn).toBeCalled();
});

上述例子会返回失败，控制台会给一警告：

![](data:image/svg+xml;utf8,)

使用Jest内置的定时器：

const fn = jest.fn()

jest.useFakeTimers()
function testTimer () {
  setTimeout(() => {
    fn()
  }, 3000)
}

test('test timer', () => {
  testTimer();
  jest.advanceTimersByTime(3000)

  expect(fn).toBeCalled();
});

useFakeTimers 就是生命在当前用例内使用Jest内置的定时器，advanceTimersByTime 是一个配置项，将控制器提前运行一段时间，简而言之，就是Jest的定时器可以让我们受控了，我们可以更加灵活的进行控制

fakeTimer这个配置项就是全局的一些配置

// jest.config.js
module.exports = {
  fakeTimer: {
    // 是否要使用全局的定时器配置(同上面的 useFakeTimers)
    enableGlobally?: boolean,
    // 定时器超前执行的时间（同上面的 advanceTimersByTime）
    advanceTimers?: boolean | number,
    // 哪些定时器的Api是不需要使用Jest内置的，具体见官网
    doNotFake?: Array<FakeableAPI>
    // 是否要用老的实现（和我们没啥关系，不用管这个配置项）
    legacyFakeTimers?: boolean,
    // 设置一个基准的系统时间戳，给内置的计时器用的，默认是Date.now()
    now?: number,
    // 暂时没看懂啥意思（先不管吧。。。。）
    tiemrLimit?: numner
  }
}

这里需要注意一小点就是，advanceTimers 这个配置项需要搭配 runAllTimers 才会生效，以上述例子：

// jest.config.js
module.exports = {
  fakeTimer: {
    enableGlobally: true
    advanceTimers: 3000
  }
}

// demo.test.ts
const fn = jest.fn()
function testTimer () {
  setTimeout(() => {
    fn()
  }, 3000)
}
test('test timer', () => {
  testTimer();
  jest.runAllTimers()

  expect(fn).toBeCalled();
});

更详细的用法和Api可以去官网详细了解

forceCoverageMatch

Array（默认值：[‘’]）

设置需要强制生成代码覆盖率报告的文件
收集代码覆盖率通常会忽略测试的文件，可以通过这个配置项覆盖这种默认行为，并在代码覆盖范围内包含其他被忽略的文件

比如，coveragePathIgnorePatterns 这个配置项配置了exclude的目录和文件，但如果想在这些排除的目录中收集某些文件的覆盖率报告，就可以利用这个配置项进行覆盖

globals

Object （默认值：{}）

可以定义一些全局变量，可以通过访问global对象拿到自定义的全局对象

如果我们定义的是引用类型的变量（对象、数组）这些，然后测试过程中改变了引用地址，新的引用是不会在其他测试文件中生效的，也就是每个测试文件中的全局变量都是引用的global的原始值

另外，配置的变量必须是可序列化的JSON格式，所以，函数就不能声明了，可以使用 setupFIles

globalSetup

String （默认值：undefined）

全局注入器，这个选项可以让我们自定义注入全局模块
可以设置一个setup文件的路径，然后该文件必须导出一个函数（同步异步都行）
这个函数会在执行用例前触发，收到Jest的两个参数：globalConfig 和 projectConfig

就是更加灵活的注入一些全局变量或者进行一些操作

比如一些函数，比如一些变量是需要异步注入的等等

// jest.config.js
module.exports = {
  globalSetup: '<rootDir>/setup.js'
}

// setup.js
const fun = () => {
  return 1
}
module.exports = async function (globalConfig, projectConfig) {
  // console.log(globalConfig);
  // console.log(projectConfig);
  globalThis.fun = fun
};

然后在用例中就能获取注入的全局模块了

![](data:image/svg+xml;utf8,)

globalTeardown

String （默认值：undefined）

全局拆卸器，这个选项可以让我们
可以设置一个teardown文件的路径，然后该文件必须导出一个函数（同步异步都行）
这个函数会在执行用例完成触发，收到Jest的两个参数：globalConfig 和 projectConfig

其实 globalTeardown 和 globalSetup 这两个配置项个人理解就是 Jest 给使用者提供的两个注入全局模块的hooks，我们可以更方便的操控一些全局模块

比如如果单测涉及到数据库，那么开始的时候我们需要注入数据库的全局变量，然后测试结束之后肯定得关闭数据库的连接，那么注入的过程就可以在setup中操作，断开连接就可以在teardown中操作

haste

Object（默认值：undefined）

Jest内置 jest-haste-map 作为文件 爬取/缓存 系统
这个配置项就是将一些参数透传给这个模块

后来直接去 jest-haste-map github地址 看了下注释的说明，才算大概理解这个配置项的意思

第一张图说明了 jest-haste-map 的作用：

当我们执行用例的时候，如果项目的文件数量非常巨大，热更新模式下，每次执行用例都去扫描文件的话是非常损耗性能的，利用 Haste Map（加速图？不知道怎么翻译，就按照英文好了）来提升性能就非常重要了

injectGlobals

Boolean（默认值：true）

是否将Jest的全局变量（例如：it test expect...）注入到 global 环境中
如果关闭，这些变量需要我们手动引入

如果关闭：

![](data:image/svg+xml;utf8,)

所以需要手动引入，官网示例代码是用esm的方式引入的，在没有进行一些配置之前，是有问题的，可以改成CommonJS的方式引入来验证：

const { it, expect } = require('@jest/globals');

it('1', () => {
  expect(3).toBe(3)
})

maxConcurrency

Number（默认值：5）

用来限制 test.concurrent 模式下的测试并发数
超过该限制的用例都会排队，并在有用例完成释放之后继续执行

前提需要大概了解下 test.concurrent 的作用，简单的来说就是可以并发进行一些异步函数的测试，举个例子：

it.concurrent("test", () => {
  return new Promise((resolve) => {
    setTimeout(() => {
      expect(1 + 2).toBe(3)
      resolve(true)
    }, 1 * 1000)
  })
}, 5000)

concurrent模式下接受三个参数，第一个同样是用例名称，第二个是一个异步函数，第三个则是超时时间，当异步函数执行的时间超过超时时间时，会报错并取消，默认是5秒：

![](data:image/svg+xml;utf8,)

可以进行一个测试，上述用例拷贝5份，然后配置 maxConcurrency 先为2，执行用例：

![](data:image/svg+xml;utf8,)

可以看到 2、4两个用例是并发执行的，并没有消耗时间，总的时常也是3s而不是5s

再把 maxConcurrency 设置为5：

![](data:image/svg+xml;utf8,)

结果显而易见了

moduleDirectories

Array（默认值：[“node_modules”]）

定义优先加载的依赖的目录路径，比如，同时在 node_modules 和 自定义一个目录 my_modules 中，同时创建一个mod的模块：

// node_modules/mod/index.js
module.exports = 1;

// node_modules/mod/package.json
{
  "name": "mod",
  "files": [
    "index.js"
  ]
}


// my_modules/mod/index.js
module.exports = 2;

// my_modules/mod/package.json
{
  "name": "mod",
  "files": [
    "index.js"
  ]
}

然后创建一个简单的用例，看看不同顺序的配置引用的模块的区别：

const a = require('mod')

it("test1", () => {
  expect(a + 1).toBe(3)
})

node_modules配置在前，由于优先使用node_modules中的mod模块，所以值为2，用例不通过：

![](data:image/svg+xml;utf8,)

![](data:image/svg+xml;utf8,)

my_modules配置在前，则优先使用my_modules中的mod模块，所以值为3，用例通过

moduleFileExtensions

Array（默认值: [“js”, “mjs”, “cjs”, “jsx”, “ts”, “tsx”, “json”, “node”]）

作用类似于webpack的extensions配置，当 import / require 没有指定文件后缀时，Jest将从上述配置的数组中从左到右查找扩展名

// 1.test.tsx
const add = require('./add')

it('test', () => {
  expect(add(1, 2)).toBe(3);
});

// add.tsx
module.exports = function(x, y) {
  return x + y;
}

// jest.config.js
module.exports = {
  moduleFileExtensions: ["js"]
};

执行用例，会提示找不到找不到用例：

![](data:image/svg+xml;utf8,)

// jest.config.js
module.exports = {
  moduleFileExtensions: ["js", "tsx"]
};

则运行正常：

![](data:image/svg+xml;utf8,)

moduleNameMapper

Object（默认值：null）

一个 从正则表达式到允许存根资源的模块名称或模块名称数组 的映射，
比如单独的图片或者样式模块

简而言之就是对一些模块的代理处理

以下面的例子来说，我们先定义一个模块，为了好区分正则，以cjs后缀结尾：

// origin.cjs
module.exports = {
  a: 1
};

然后写一个简单的用例测试下：

const obj = require('./origin.cjs');

it("test1", () => {
  expect(obj.a).toBe(1);
})

这个时候我们再写一个处理函数，作用也很简单，就是当我们获取对象某个key的值的时候，返回的就是这个key：

// idObj.js
var idObj = new Proxy({}, {
  get: function getter(target, key) {
    return key;
  }
});

module.exports = idObj;

再配置一下moduleNameMapper:

// jest.config.js
module.exports = {
  moduleNameMapper: {
    '\\.(cjs)$': '<rootDir>/moduleNameMapper/idObj.js',
  }
};

当使用cjs后缀的模块时，都使用idObj代理处理一下。这个时候再执行原来的用例就会报错了，实际上被处理了之后，断言的值应该是key值，也就是a，可以自行验证一下

By the way，上述那个idObj.js，实际上就是 identity-obj-proxy 的源码简易版

这个配置项作用的话就是加快用例的执行速度，一些模块比如图片、样式啥的在执行用例的时候并不需要真正去加载，代理一下即可

modulePathIgnorePatterns

Arrary （默认值： []）

在这些路径之前所有与模块路径匹配的正则字符串数组对模块加载器被认为是 “可见的”。
如果给定的模块路径与任何模式匹配，则它在测试环境中将不支持被 require()

功能上来说就是可以忽略一些目录下的测试文件，同样举个例子：

// index.test.tsx
it("test1", () => {
  expect(1).toBe(1);
})

// index1.test.tsx
it("test2", () => {
  expect(2).toBe(2);
})

// jest.config.js
module.exports = {

  modulePathIgnorePatterns: [
    '<rootDir>/modulePathIgnorePatterns/index.test.tsx'
  ]
};

执行用例，就只会执行index1这个用例。

但是后面还有一个配置项 testPathIgnorePatterns 也是相同的功能，目前不是很清楚两者的主要区别是啥。

看了一下主要的描述区别是，testPathIgnorePatterns是skip，而 modulePathIgnorePatterns 的描述是：

![](data:image/svg+xml;utf8,)

modulePaths

Array（默认值：[]）

一个可选的用来设置 NODE_PATH 环境变量的配置
值为一个用来表示需要模块额外去查找的路径的绝对路径的数组

node默认引入模块是去查找node_modules下的模块，这个配置项可以配置一些别的查找路径。简单举个例子：

// 根路径下建立目录 customModule/index.ts
module.exports = 1;

// index.test.ts
const customModule = require('customModule');

it("test1", () => {
  expect(customModule).toBe(1);
})

node_modules中并没有customModule这个模块，这个时候require肯定是报错的。加上配置：

// jest.config.js
modulePaths: [
   '<rootDir>/'
]

告诉Jest还可以从根路径下找找，再跑一遍用例，正常pass

notify

Boolean （默认值：false）

是否激活系统的提示，需要额外安装依赖 node-notifier

要注意的就是需要去开启通知权限，以mac系统为例，要开启terminal-notifier的通知权限

![](data:image/svg+xml;utf8,)

notifyMode

String （默认值：failure-change）

如果 nofity 配置项开启，可以通过这个依赖项配置什么情况下会进行系统通知
默认有如下模式：
- always：总是会发
- failure: 用例测试失败时候发
- success: 用例成功的时候发
- change: 用例状态改变的时候发
- success-change: 用例通过或者失败的时候发
- failure-change: 用例失败或者通过的时候发

resolver

String（默认值：undefined）

这个配置项允许我们自定义一个模块解析器，满足以下两个条件中任何一个即可：
1、一个函数，接受两个参数，第一个参数为字符串（需要处理的路径），第二个参数为一个配置对象。
函数需要返回模块的路径，如果模块没有找到则需要抛出错误

2、一个对象，包含 async / sync 中的至少一个属性。
sync 属性值为第一点中描述的函数
async 属性值同理，只不过最终的结果需要resolve，异常需要reject

我们可以按照第一点来写一个简单的模块解析器，先进行准备工作：

// mods/mod.ts
module.exports = 1

// index.test.tsx
const mod = require('mod')

it("test1", () => {
  expect(mod).toBe(1);
})

我们就这样执行用例肯定是会报错的，因为node_modules中并没有mod模块，我们的mod模块在同级的mods目录下，所以此时我们自定义一个resolver：

// resolver.js
const path = require('path')

module.exports = function(_path, options) {
  // 当加载到mod时使用处理后的路径
  if (_path === 'mod') {
    return path.join(__dirname, 'mods/mod.ts')
  }
  
  // 其他情况使用默认的解析器处理
  return options.defaultResolver(_path, options)
}

// jest.config.js
module.exports = {
  resolver: '<rootDir>/resolver.js'
};

重新执行一次，mod模块就被正常加载了，算是最最基础版的一个模块解析器了

另外options配置对象中，除了defaultResolver 这个默认解析器的配置外，还有一些其他的数据，需要详细了解的话可以参考下哈

snapshotResolver

String（默认值：undefined）

可以解析测试<->快照路径的模块的路径
此配置选项允许您自定义 Jest 在磁盘上存储快照文件的位置

这个配置项我们得先大概了解一下快照测试，简单来说就是：

当我们对一个组件进行快照测试的时候，第一个版本运行完单测，会在本地生成一份这个组件的快照文件。当我们对这个组件改动重新运行单测后，jest就会和上一份的快照对比，如果本次改动导致了快照发生了变化，jest就会报错并提示

**快照测试个人理解其实是更注重对比提醒而不是逻辑纠错的作用，**它能让我们可以知道某次改动是否对组件的渲染结果造成了影响

看一个例子：

// Link.tsx
import React, {useState} from 'react';

export default function Link() {
  const [status, setStatus] = useState(1);

  const href = status === 1 ? 'Href1' : 'Href2'
  const text = status === 1 ? 'Link1' : "Link2"

  return (
    <a href={href}>{text}</a>
  );
}

// 1.test.tsx
import React from 'react';
import renderer from 'react-test-renderer';
import Link from './Link';

it('renders correctly', () => {
  const tree = renderer
    .create(<Link />)
    .toJSON();
  expect(tree).toMatchSnapshot();
});

运行，由于第一遍没有快照，所以会在同目录下生成一份 __snapshots__/1.test.tsx.snap 的一份快照：

// Jest Snapshot v1, https://goo.gl/fbAQLP

exports[`renders correctly 1`] = `
<a
  href="Href1"
>
  Link1
</a>
`;

此时如果我们需要对组件进行改动，但不小心改动了status初始的值：

// Link.tsx

import React, {useState} from 'react';

export default function Link() {
  const [status] = useState(2);

  const href = status === 1 ? 'Href1' : 'Href2'
  const text = status === 1 ? 'Link1' : "Link2"

  return (
    <a href={href}>{text}</a>
  );
}

这个时候实际上是对组件造成了破坏性的改动，但我们本身并不知道，再执行一遍用例，jest就会和本地的快照进行比对：

添加图片注释，不超过 140 字（可选）

通过这个diff过程，就能很清楚知道这次的改动造成的影响。而如果改成2确实是需要的，那就说明上次的快照需要更新，执行一下 yarn test - u 即可更新快照

上述就是对快照测试的一个简单的介绍，然后这个snapshotResolver配置项就是可以配置快照文件的生成路径，例如在上述例子中，默认生成快照文件的路径是 /tests/__snapshots__/1.test.tsx.snap

如果我们要将快照生成在 /tests/snapshots 目录下，就可以进行如下配置：

// jest.config.js
module.exports = {
  snapshotResolver: '<rootDir>/resolver.js'
};

// resolver.js
module.exports = {
  /**
   * 处理快照文件路径
   * @param testPath - 测试文件路径
   * @param snapshotExtension - 快照文件后缀（一般为.snap）
   * @returns 快照文件路径
   */
  resolveSnapshotPath: (testPath, snapshotExtension) => {
    return testPath.slice(0, testPath.lastIndexOf('/')) + '/snapshots' + testPath.slice(testPath.lastIndexOf('/')) + snapshotExtension
  },

  /**
   * 处理测试文件路径
   * @param snapshotFilePath - 快照文件路径
   * @param snapshotExtension - 快照文件后缀（一般为.snap）
   * @returns 测试文件路径
   */
  resolveTestPath: (snapshotFilePath, snapshotExtension) => {
    return snapshotFilePath.replace('/snapshots', '').slice(0, -snapshotExtension.length)
  },

  // 一个用来预检resolver功能的路径例子（方便log来测试自己的路径对不对，会被resolveSnapshotPath调用）
  testPathForConsistencyCheck: 'tests/example.test.js',
};

具体的描述已在代码中标注，应该是很详细了，顺带一提这三个属性都是必填，缺失会报错

transform

Object<string, pathToTransformer | [pathToTransformer, object]>（默认值：{“\.[jt]sx?$”: “babel-jest”}）

transform配置是一个对象，用于定义在运行测试时，如何转换测试文件。

"\\.[jt]sx?$"：键名表示要转换的文件的文件名的匹配模式。
"babel-jest"：键值表示使用的转换器的名称

如果你在Jest的配置文件中使用了transform配置，那么Jest在运行测试时会使用babel-jest转换器来转换所有符合"^.+\.jsx?$"这个匹配模式的文件。
使用transform配置可以帮助你在运行测试时转换你的代码，从而可以使用最新的JavaScript语法进行测试

我们可以利用后面的一个配置项 - transformIgnorePatterns 来进行简单的测试

transformIgnorePatterns这个配置项配置的是将一些文件忽略，不使用transform的转换器进行转换

// transform/mod.js
export default 1

// transform/index.test.tsx
import mod from './mod';

it("test1", () => {
  expect(mod).toBe(1);
})

// jest.config.js
module.exports = {
  transform: {
    '^.+\.(js|jsx|ts|tsx)$': 'babel-jest',
  },

  transformIgnorePatterns: [
    '<rootDir>/transform/'
  ]
};

我们将transform下的模块不使用babel-jest，执行用例，会报错：

![](data:image/svg+xml;utf8,)

因为mod模块是使用es模块导出的，而jest默认支持的是commonJS的规范，所以报错告诉我们不支持es的语法（当把mod换成commonJS规范导出之后，是没问题的）

然后我们把 transformIgnorePatterns 中的配置删除，mod.js 也走babel-jest进行转换，重新执行一遍用例，就正常了

Jest默认使用babel-jest作为转换器，当然也可以参考这一章来实现一个自定义的 transformer：Code Transformation · Jest

// sequencer.js
const Sequencer = require('@jest/test-sequencer').default;

class CustomSequencer extends Sequencer {
  /**
   * 通过命令行 --shard=shardIndex/shardCount 来决定执行哪部分的测试文件
   * 比如 --shard=2/3，意思就是将用例分成3份，执行第二份
   * shard函数在sort函数前执行
   */
  shard(tests, {shardIndex, shardCount}) {
    const shardSize = Math.ceil(tests.length / shardCount);
    const shardStart = shardSize * (shardIndex - 1);
    const shardEnd = shardSize * shardIndex;

    return [...tests].slice(shardStart, shardEnd);
  }

  /**
   * 对shard执行后的测试文件进行排序，以确定执行顺序
   * sort函数在shard函数后执行
   */
  sort(tests) {
    const copyTests = Array.from(tests);
    const resultTests = copyTests.sort((testA, testB) => (testA.path.length > testB.path.length ? -1 : 1));
    return resultTests;
  }
}

module.exports = CustomSequencer;

具体的说明在注释中已经写的比较详细了，此时shard函数的作用如注释，就是执行命令行中指定部分的用例，sort函数就是按测试文件的名称长度进行执行

testMatch

Jest用来检测测试文件的全局匹配表达式。
默认情况下是获取 __tests__ 目录下的 js/ts/jsx/tsx 文件，以及任何后缀中带有 .test/.spec 的文件
也会获取 test.js/spec.js

配置哪些文件你是想进行测试的

testRegex

作用同testMatch，不过两个配置不要同时定义，否则会报错

testTimeout

Number（默认值：5000）

https://github.com/facebook/jest/issues/9696

unmockedModulePathPatterns

Array（默认值）

一组用来匹配那些 被引入但是不需要被自动mock的模块的路径的 正则表达式
如果某个模块匹配了其中的一个规则，那么就不会被自动mock

这个配置项其实场景比较少，因为一般来说我们很少会开启 automock，如果有需要mock的模块，一般也会是在各个测试文件的内部去mock。

// mod.ts
module.exports = () => 1

// mod1.ts
module.exports = () => 2

// index.test.tsx
const mockFn = require('./mod')
const mockFn1 = require('./mod1')

mockFn.mockReturnValue(3)
mockFn1.mockReturnValue(4)

it("test1", () => {
  expect(mockFn()).toBe(3);
  expect(mockFn1()).toBe(4);
})

// jest.config.js
module.exports = {
  automock: true
};

由于开启了automock，所以此时执行用例是完全正常的。然后我们配置一下unmockedModulePathPatterns，将mod1排除：

// jest.config.js
module.exports = {
  automock: true,
  unmockedModulePathPatterns: [
    '<rootDir>/unmockedModulePathPatterns/mod1.ts'
  ]
};

watchPathIgnorePatterns

Array （默认值：[]）

一个正则表达式的路径数组，用来匹配那些不需要在 watch 模式下被重复执行的测试文件
如果文件路径符合该数组中的某个规则，当这些文件更新的时候不会触发热重载

这些规则是匹配绝对路径的，可以使用 <rootDir>用来表示根路径

当然这个配置项只在 --watch 的模式下才会生效

watchPlugins

Array<String ｜ [String, Object]> （默认值：[]）

自定义使用的监听文件变化的插件数组

用默认的就好了，具体有哪些插件可以去官网中查看

watchMan

Boolean（默认值：true）

是否使用watchman对文件的更改进行监听

总结

这篇文章写的对自己使用配置的时候的疑惑点都写了比较详细的补充，有些会用例子来说明下（貌似都喜欢看简洁的文章哈哈）

单测说实话确实是吃力不讨好的事情，一没什么技术难度，二也不会对产出有很大的显性帮助，更难的是个人感觉写单测有个前提条件就是需要团队稳定，因为单测也是需要去维护的。很多时候当业务逻辑更改的时候，单测也要同时去修改。变动比较频繁的话，单测就会被遗忘，逐渐没人维护，导致和业务代码偏移越来越大，最后就成鸡肋

但这玩意儿在后面代码重构或者涉及到大面积改动的时候，太有用了。而且一开始不写，后面再补是非常麻烦的，有时候甚至得先重写下代码才行。所以如果条件允许，还是建议写了
